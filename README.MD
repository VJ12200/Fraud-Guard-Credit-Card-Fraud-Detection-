## Overview

FraudGuard is a full-stack fraud detection system that processes credit card transactions through two independent layers:

1. **Rule Engine (Java)** вҖ” Acts as the first layer that contains hardcoded rules that check for fraud 
2. **ML Model (Python)** вҖ” XGBoost classifier trained on 1.85M transactions with 15 engineered features, served via FastAPI

Both layers run on every transaction. If the Python service is unavailable, the rule engine continues operating independently вҖ” the system never crashes.

Transactions can be submitted as a **single object or a JSON array** through the same endpoint, and are evaluated for high-amount violations, structuring, improper frequency, graph-based money laundering cycles, and ML-detected anomalies simultaneously.

## Features

### Rule Engine
- **High Amount Rule** вҖ” flags any single transaction exceeding $1,000,000
- **Structuring Rule** вҖ” monitors cumulative spend per sender over a rolling 1-hour window; flags when total exceeds $1,000,000 (detects split-transaction laundering)
- **Frequency Rule** вҖ” flags senders with more than 10 transactions in the last hour
- **Graph Cycle Detection** вҖ” builds a directed transaction graph and runs DFS from every node to detect money laundering rings (AвҶ’BвҶ’CвҶ’A patterns)

### ML Model
- XGBoost classifier trained on 1.85M synthetic transactions
- Hyperparameters tuned via Optuna (50 trials, TPE sampler, maximising PR-AUC)
- SMOTE oversampling on training fold only to handle 0.58% fraud rate
- `scale_pos_weight` set to neg/pos ratio as additional imbalance correction
- Threshold tuned post-training via precision-recall curve analysis

### Feature Engineering (computed at inference, no user input required)
| Feature | Description |
|---|---|
| `hour` | Hour of day extracted from transaction timestamp |
| `time_since_last_txn` | Seconds since same card's previous transaction |
| `txn_count_per_card_day` | Number of transactions this card made today |
| `amt_vs_card_mean` | Signed deviation from this card's historical average spend |
| `amt_vs_card_std` | Z-score-like ratio against card's spend standard deviation |
| `geo_distance` | Euclidean distance between cardholder and merchant location |

### API Design
- Single endpoint accepts both `{...}` and `[{...}, {...}]` вҖ” no separate batch route
- Jackson `readTree` parsing handles type detection at runtime
- Per-transaction try/catch in batch processing вҖ” one failure never crashes others
- `@JsonIgnoreProperties(ignoreUnknown = true)` on model prevents unknown field errors

### UI
- Thymeleaf frontend with dark terminal aesthetic
- Single and batch transaction modes with toggle
- Category dropdown with all 13 dataset categories
- Auto-computed feature note shown to users
- Colour-coded results (green OK, red FLAGGED, yellow FAILED)

## Quick Start

### Prerequisites
- Java 17+
- Python 3.9+
- Maven 3.8+

### 1. Clone the repository

```bash
git clone https://github.com/yourusername/fraudguard.git
cd fraudguard
```

### 2. Download the dataset

See [data/README.md](data/README.md) for instructions.
Place `fraudTrain.csv` in the `data/` directory.

### 3. Set up Python environment

```bash
pip install -r requirements.txt
```

### 4. Train the ML model

```bash
cd python

# Run once вҖ” finds best hyperparameters via Optuna (~30вҖ“60 mins)
python tune.py

# Fast retraining using saved params (~5 mins)
python train.py

# Evaluate model and save decision threshold
python evaluate.py
```

### 5. Start the Python inference API

```bash
cd python
uvicorn app:app --reload --port 8000
```

Verify: `http://localhost:8000/health`

Expected response:
```json
{
  "status": "ok",
  "features_expected": 15,
  "threshold": 0.6856
}
```

### 6. Configure Spring Boot

```bash
cd java
cp src/main/resources/application.properties.example \
   src/main/resources/application.properties
```

Edit `application.properties` with your database settings if not using H2.

### 7. Start Spring Boot

```bash
cd java
mvn spring-boot:run
```

### 8. Open the UI

```
http://localhost:8081/ui
```

## Rule Engine

Rules are Spring `@Component` beans implementing the `Rule` interface. `RuleEngine` autowires `List<Rule>` вҖ” adding a new rule requires only creating a new `@Component` class, no changes to `RuleEngine`.

```java
public interface Rule {
    boolean apply(FinancialTransaction t);
    String getName();
}
```

### HighAmountRule
Fires when `amount > 1,000,000`. Straightforward threshold check.

### StructuringRule
Queries all transactions from the same sender in the last hour (excluding current), sums their amounts. Fires when cumulative total exceeds $1,000,000. Detects the common money laundering technique of splitting large transfers into smaller amounts below reporting thresholds.

### FrequencyRule
Queries transactions from the same sender in the last hour. Fires when count вүҘ 10. Detects card testing and rapid-fire fraud attacks.

### Graph Cycle Detection
`GraphService` builds a directed graph where each edge represents a transaction (sender вҶ’ receiver). `CycleDetection` runs DFS from each sender node. If a path returns to the origin through 3+ nodes, a money laundering ring is detected. Graph is cached with a 5-minute TTL and invalidated after each batch save.

## Risk Scoring

All violations are scored and summed, capped at 100.

| Rule | Score |
|---|---|
| High Amount Rule | +50 |
| Structuring Rule | +40 |
| Graph Cycle Fraud | +70 |
| High Frequency Rule | +30 |
| AI Anomaly Detection | +60 |
| Amount > $2,000,000 | +10 bonus |
| **Maximum** | **100** |
